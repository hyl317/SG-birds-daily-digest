# SG Birds Daily Summary

A set of Python scripts that read the "SG BIRDS (Sightings/Live Update)" Telegram group, summarize bird sightings using Claude Haiku, and provide a searchable archive via a Telegram DM bot. The bot also falls back to the eBird API for locations outside Singapore.

## Architecture

Three processes share one SQLite database (`sightings.db`):

1. **`sg_birds_summary.py`** (scheduled via launchd) — fetches recent messages, calls Claude Haiku to extract a structured sightings list + an HTML digest, sends the digest to Telegram, and persists sightings to SQLite. Also prunes rows older than 90 days at the start of each run.

2. **`bot.py`** (long-running via launchd KeepAlive or systemd on the Hetzner VM) — separate Telegram bot account (BotFather token). Handles private DMs: text queries go through geocoding-based routing (SG → local FTS5 search, non-SG → eBird `/obs/geo/recent`), and shared location pins go straight to eBird at those coordinates. Inline-query mode has been removed.

3. **`backfill.py`** (one-time) — fetches the last 90 days of group history in 1-day chunks, extracts sightings via Claude, and populates the DB.

## Key Files

- `sg_birds_summary.py` — Main scheduled script (fetch, summarize, send, persist)
- `bot.py` — Long-running DM bot (local DB + eBird fallback + GPS pin support)
- `ebird.py` — Photon forward geocoding, Nominatim reverse geocoding, and eBird API wrapper used by `bot.py`
- `classify.py` — Pure-function query classifier (species / location / ambiguous) used by `bot.py` to route text queries
- `taxonomy.py` — Downloads and caches the eBird taxonomy; builds the lookup sets consumed by `classify.py`
- `backfill.py` — One-time historical import
- `db.py` — SQLite schema (sightings table + FTS5 virtual table) and query helpers
- `acronyms.md` — Known acronyms for bird species, locations, and terms. Claude uses this to expand abbreviations. New acronyms are auto-detected and appended by the script. Gitignored — each host (Mac, Hetzner) maintains its own copy.
- `config.json` — Runtime config (group ID, schedule). Generated by interactive setup on first run. Gitignored.
- `group_config.json` — Static group metadata. Used by `bot.py` to build deep links to original messages.
- `sightings.db` — SQLite archive (90-day rolling window). Gitignored.
- `taxonomy.json` — Cached eBird taxonomy (~6 MB, 18k species). Fetched by `taxonomy.py` on startup. Gitignored — per-host.
- `.env` — API keys: `TELEGRAM_API_ID`, `TELEGRAM_API_HASH`, `ANTHROPIC_API_KEY`, `BOT_TOKEN`, optional `EBIRD_API_KEY` (get one at https://ebird.org/api/keygen — enables the non-SG fallback). Never commit.
- `com.hyl.sgbirds-summary.plist` — launchd schedule for the daily summary
- `com.hyl.sgbirds-bot.plist` — launchd config for the always-on bot (KeepAlive=true)
- `session/` — Telethon session files (`sg_birds` for the userbot, `sg_birds_bot` for the bot)

## Dependencies

- `telethon` — Telegram client (used by both the userbot and the DM bot)
- `anthropic` — Claude API (uses `claude-haiku-4-5-20251001`)
- `python-dotenv` — Env file loading
- `requests` — HTTP client for Nominatim + eBird
- `sqlite3` (stdlib) — storage layer, FTS5 for search

## Commands

```bash
# Install dependencies
pip install -r requirements.txt

# First run of the summary script (interactive setup)
python sg_birds_summary.py

# Re-configure
rm config.json && python sg_birds_summary.py

# Backfill 90 days of history into sightings.db (one-time)
python backfill.py

# Run the inline-query bot (foreground, for testing)
python bot.py

# Install the bot as a launchd service (always-on)
ln -sf "$PWD/com.hyl.sgbirds-bot.plist" ~/Library/LaunchAgents/
launchctl load ~/Library/LaunchAgents/com.hyl.sgbirds-bot.plist

# Check status
launchctl list | grep sgbirds

# View logs
cat /tmp/sg_birds_summary.log
cat /tmp/sg_birds_bot.log
```

## Bot setup (one-time)

1. Talk to @BotFather in Telegram, run `/newbot`, follow prompts, copy the token
2. Add `BOT_TOKEN=<token>` to `.env`
3. (Optional) Add `EBIRD_API_KEY=<key>` to `.env` to enable the non-SG eBird fallback
4. Run `python bot.py` once manually to test, then load the launchd plist (Mac) or systemd unit (server)

The bot does NOT need to be added to the source group. Users just DM it privately.

## Notes

- First run of the userbot must be manual for Telegram phone-number login.
- `acronyms.md` is both human-editable and machine-appended. Manual edits take precedence — the script only appends new entries.
- Claude is prompted to return its response in two parts: a fenced ` ```json ``` ` block with structured sightings, followed by the HTML summary. If JSON parsing fails, the script logs a warning and skips the DB write but still sends the digest.
- The summarization prompt explicitly instructs Claude to only report species named in text (not inferred from photos), and to skip species mentioned only in queries/discussion without an actual sighting.
- Messages are capped at 600K chars before sending to Claude.
- The DB is pruned to a rolling 90-day window at the start of each summary run.
- The bot's routing for text queries is classifier-driven. `classify.py` labels each query as `species`, `location`, or `ambiguous` using lookup sets built by `taxonomy.py` from the eBird taxonomy. `species` → local SQLite archive only. `location` → Photon geocoder → eBird (or local archive if the top candidate is inside the SG bounding box 1.15–1.48 / 103.6–104.1). `ambiguous` → local archive first, Photon fallback if empty. Shared Telegram location pins always bypass the classifier and go straight to eBird.
- Forward geocoding uses Photon (`photon.komoot.io`) — free, no API key, much better full-text indexing than Nominatim. Reverse geocoding (used only for the GPS-pin reply header) stays on Nominatim.
- `taxonomy.py` caches the eBird taxonomy to `taxonomy.json` on disk (gitignored, ~6 MB, 18k species). Refreshed after 30 days. Needs `EBIRD_API_KEY` for the first download; after that, the cache is used even if the key is unset. Gitignored per-host, like `acronyms.md`.
